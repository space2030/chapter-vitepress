如何处理千万级别mysql数据库锁表问题？ #

01 先搞清楚mysql有哪些存储引擎？ #

MyISAM

支持表级锁，无论是insert、update、delete都会对整张表加锁，该存储引擎不支持事务，性能比较好，但由于整表锁，tps不太高，不适合高并发操作。

Innodb 支持表锁和行锁，该存储引擎支持事务，由于支持细粒度的行锁，比较适合高并发操作

什么情况下使用表锁

  1. sql未使用索引，更新或删除单表中的数据；
  2. sql使用索引，但不是唯一索引，这个时候如果更新表中的大量数据或全量数据，此时如果使用行锁，会造成事务长时间等待、锁冲突，因此mysql会升级锁表sql操作几张表，即大事务，比较复杂，这个时候mysql为了防止死锁和事务回滚，会同时锁相关的表。
  3. 使用sql语句显示锁表。类似加：for update;
  4. 对线上大数据量并且活跃时间长的表直接进行DDL操作或加索引操作，这也会造成锁表。

02 如果线上环境遇到锁表后怎么办？ #

  • a. 当线上如果发现大量sql语句导致长时间锁表，可以通过show processlist命令找出锁表的进程号。然后执行kill进程号命令，快速结束锁表语句。
  • b. 根据上一步的锁表sql快速从程序中找出是哪个业务导致的，解决bug后紧急发版，如果是非核心业务导致的，为了保证系统不被拖垮，可以将接口做降级处理。
  • c. 可以写一些检查锁表的运维脚本，每隔几分钟检查一次，出现长时间锁表的sql发出告警提醒。

03 如何避免锁表的情况？ #

  • a. 对于大表的操作，查询条件一定要保证命中索引，如果能命中唯一索引就更好了。
  • b. 我们在程序开发的时候，尽可能将大事务拆分为小事务，减少锁表或者回滚，比如：抽离部分业务逻辑异步发送消息队列处理。
  • c. 更新很频繁的业务，尽量转为批量处理，另外尽可能减少单次处理的数据量，大数据量可以分批处理。
  • d. 不要通过select * from user for update显示锁表，高并发的时候会严重影响tps，通过乐观锁加版本号实现或者先写到缓存然后异步写表等方案解决。
  • e. 业务高峰期，不要随便直接加字段或者索引，尽量用户不活跃的时候执行，但是如果一定要在高峰期操作，可以复制旧表结构创建一个新表，然后在新表上加字段或索引，接着将原来旧表的数据复制到新表，再rename两个表，最后将rename期间旧表的增量数据迁移到新表。
  • f. 采用读写分离架构。

锁表的几种场景

  1. 采用 MyISAM 存储引擎
  2. sql 未使用索引，更新或删除单表中的数据
  3. sql 使用索引，但不是唯一索引，更新表中的大量数据或全量数据
  4. 使用类似 select * from user for update 的 sql 语句显示锁表
  5. 对线上大数据量并且活跃时间长的表直接进行 DDL 操作或加索引操作
