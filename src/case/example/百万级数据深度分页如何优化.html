百万级数据深度分页如何优化 #

01 场景举例 #

场景1. 查询文章列表，一直滑动翻页，不用跳转到指定页数。 场景2. 从数据库查询百万客户数据写入到redis。 场景3. 访问某小程序的积分商城查看商品，一直滑动翻页，不用跳转到指定页数。

02 问题分析 #

深度分页问题SQL

select
c.id,c.no,c.name
from
t_customer c
where
c.delete_flag = 0
order by c.id limit #{offset}, #{pageSize}

深度分页造成的结果，当以上的场景翻页很深，offset越来越大，回表的记录越来越多， sql查询性能急剧下降，会出现大量的慢SQL（超过1s的）。

03 解决方案 #

  1. 子查询，利用聚集索引减少回滚，从而提升查询效率，类似以下的SQL：
select
c.id,c.name,c.no
from
t_customer c
where
c.id >= (select id 
         from t_customer 
         where delete_flag = 0 
         ORDER BY id 
         limit #{offset},1) 
limit #{pageSize};
  1. 内连接（inner join）查询，我们可以把子查询（利用聚集索引）的结果当成一张临时表，然后和原表进行关联查询，类似以下的SQL
select
c.id,c.name,c.no
from
t_customer c,(select id 
                from t_customer 
                where delete_flag = 0 
                ORDER BY id 
                limit #{offset},#{pageSize}) as d
where
c.id = d.id;
  1. 使用游标分页查询，当我们查询第二页的时候，把第一页的查询结果（返回最后一条记录的ID）放到第二页的查询条件中，类似以下SQL
select c.id,c.no,c.name
from t_customer c
where c.deletee_flag = 0 and id > #{lastId} 
order by c.id limit #{pageSize};

总结

  1. 基于c端的分页查询，解决场景一（1）（3），翻页很深性能差的问题：三（3）
  2. 从数据库查询百万员工数据写入redis
    • 深度分页查询、防止产生大量的慢SQL
    • 利用redis的pipline批量写入数据到redis
    • 利用TimeUnit.MILLISECONDS.sleep(500)，防止短时间写命令太高，导致redis cpu飙升。
